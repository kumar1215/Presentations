Recommended Add-On: GitHub Advanced Security

GHAS is the only paid add-on we recommend. It addresses our most critical gap — we have had zero security scanning since August 2022.

🔍

Code Security — CodeQL SAST

  • ✓ Static analysis with 2,000+ security queries for C#, TypeScript, JavaScript
  • ✓ Zero-config default setup — auto-detects languages, runs on every PR
  • ✓ Deep semantic analysis — tracks data flow through ASP.NET MVC, Entity Framework, and DI patterns
  • ✓ Copilot Autofix — AI-generated fix suggestions for every alert (leverages our existing Copilot license)
  • ✓ Dependency Review — blocks PRs introducing vulnerable packages
🔐

Secret Protection

  • ✓ Secret scanning on private repos — 200+ secret types detected
  • ✓ Push Protection — blocks secrets before they enter Git history
  • ✓ Auto-revocation — notifies Azure, AWS, GitHub to revoke leaked tokens
  • ✓ AI credential detection — catches hardcoded passwords that regex misses
  • ✓ Custom patterns — define Bunzl-specific secret formats
📊

Security Overview Dashboard

Organization-level view aggregating all security alerts across every repo. Prioritization by severity and exploitability. Shows the top 10% most critical alerts first. Provides auditable evidence for compliance (SOC 2, ISO 27001, customer security questionnaires).

Why push protection matters
Without push protection, a committed secret is already in Git history by the time it's detected. Removing it requires force-pushing to rewrite history, rotating the compromised credential, and assessing whether it was exposed during the window between commit and detection. GHAS push protection prevents this entirely — the secret never reaches the repository.

What about free alternatives?

Open-source tools like Semgrep (SAST) and Gitleaks (secret scanning) exist, but carry significant trade-offs:

Limitations of free SAST (Semgrep)

  • ✗ Pattern-based only — weaker at tracking data flow across .NET method boundaries vs CodeQL's semantic analysis
  • ✗ Higher false positives — requires ongoing rule tuning to prevent alert fatigue
  • ✗ No auto-fix — developers manually research CVEs and write fixes
  • ✗ No centralized dashboard — per-repo alerts only
  • ✗ Ongoing maintenance — monthly version updates, rule tuning, scan troubleshooting
  • ✗ Community support only — no vendor SLA on detection coverage

Limitations of free secret scanning (Gitleaks)

  • ✗ No push protection — secrets detected after commit, already in Git history
  • ✗ No auto-revocation — manual credential rotation per incident
  • ✗ Regex-only — cannot catch unstructured credentials (passwords, connection strings)
  • ✗ Manual pattern updates — new secret types go undetected until added
  • ✗ Each leak triggers incident response: history rewrite + rotation + exposure assessment
  • ✗ Harder to defend in compliance audits — no vendor-backed SLA
The hidden cost of "free"
Free tools require ongoing DevOps investment — setup, maintenance, tuning, and incident response. They deliver weaker .NET coverage, carry no vendor SLA, and critically lack push protection. The operational overhead of maintaining free tools narrows the real savings considerably compared to the GHAS license.
